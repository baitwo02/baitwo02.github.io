# AxVisor 项目日志

在项目阶段，主要完成了设备模型的构建，以及设备注册机制的完善，通过工厂函数来注册每个设备，并且通过链接脚本来收集每个设备自身的工厂函数，以到达不需要修改核心代码就能注册设备的能力。

## 对于项目后续的规划

首先我希望将之前项目阶段的内容再完善一下，因为开发基于的版本比较老，r最后也没有提交一个很完善的 pr，由于冲突的比较多，也造成了合并的难度很大。后续我希望就是将我之前的工作 rebase 到当前的 dev 分支上，看一下能不能提交一个 pr 对项目做出贡献（~~其实是想混一个 pr~~）。

然后就是我希望可以参与到 Axvisor 对 Linux Guest 支持完善的工作当中。

我本人是一个 linux 用户，我日用的设备也是一个 win/linux 双系统的机器，之前接触到 usr 不可变系统的设计，也通过 [particleos](https://github.com/systemd/particleos) 构建了一个基于 archlinux 的一个 usr 不可变系统的镜像，虽然我最后没有将我自己使用的系统迁移到 usr 不可变系统上。当时是对与 linux 的启动流程并不是特别的清晰，而我只是使用 particleos 构建了一个 qemu 可以启动的虚拟机镜像，而虚拟机镜像和物理机所需要的镜像完全不一致，很多工作都由 mkosi 打包好并且交给 qemu 处理，所以对于文件系统的挂载以及启动所需要的 initfs 应该怎么配置，我都没有弄得很明白。

但我认为这一类基于一个配置文件来构建一个确定的系统镜像，可以天然的适配某些任务，并且天然的具备可复现能力，无论对于日常使用的稳定行，还是项目需要的可复现环境都是具有可观的优势的。

所以我希望就是可以在这个阶段能够在 AxVisor 上尽量完善 linux 客户机的启动，首先就是对于虚拟设备的完善，尽量将启动一个 linux guest 所必要的虚拟化设备给实现了。然后就是对于中断控制的完善，但这一块在项目阶段就没有做的很好，所以后续还需要在研究一下。

希望可以借此完善我对于 linux 设备启动流程的理解。
